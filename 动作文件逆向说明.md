# .rob 动作文件逆向说明

## 1. 结论概览

这个项目里的 `.rob` 文件并不是脚本源码，而是 `ACT-40` 动作容器。

目前已经可以确认：

1. 所有 `.rob` 文件都使用相同的外层容器结构。
2. 容器内部存在两种帧体格式：
   - 明文格式
   - `EYPT` 变体
3. 明文格式已经可以稳定解析到字段级。
4. `EYPT` 变体已经被上位机程序集成功解开，可以稳定恢复成明文 `ACT-40` 帧数据。

## 2. 文件外层结构

每个 `.rob` 文件都满足下面的长度公式：

`文件总长度 = 16 + 帧数 * 248`

对应结构：

- 文件头：16 字节
- 帧数组：每帧固定 248 字节

### 2.1 文件头

文件头前 16 字节含义已经基本明确：

- `0x00..0x05`：ASCII 字符串 `ACT-40`
- `0x06..0x07`：小端 16 位帧数
- `0x08..0x0B`：类型标签
  - 明文文件通常为全 0
  - 加密/编码变体为 `EYPT`
- `0x0C..0x0D`：固定值 `0x0002`
- `0x0E..0x0F`：保留字段，当前样本中为 `0x0000`

## 3. 明文帧格式

明文文件的单帧结构可以确定如下：

### 3.1 帧头 8 字节

- `0x00..0x01`：小端 16 位时长
- `0x02..0x03`：固定标记 `0x5555`
- `0x04..0x05`：保留字段，当前样本中为 `0x0000`
- `0x06..0x07`：保留字段，当前样本中为 `0x0000`

### 3.2 帧体 240 字节

帧体可以稳定按 40 组、每组 6 字节解析：

- 每组 3 个小端 16 位值
- 总计 `40 * 6 = 240` 字节

也就是：

- 槽位数：40
- 每槽位字段数：3

### 3.3 有效槽位与填充槽位

全量样本已经验证：

- 前 16 组是有效动作槽位
- 后 24 组是填充槽位

填充槽位固定为：

- `(0x5555, 0x0000, 0x0000)`

也就是说，明文帧的真实有效动作区只有前 96 字节：

- `16 个有效槽位 * 6 字节 = 96 字节`

剩余 144 字节为 filler：

- `24 个 filler 槽位 * 6 字节 = 144 字节`

### 3.4 明文槽位数据特征

样本显示：

- 大多数动作文件中，每个有效槽位只有第一个 16 位值非零
- 少数动作文件，例如 `0号立正.rob`，三个字段都可能非零

因此当前可确认：

- 第一个 16 位值很像目标舵机位置
- 第二、第三个 16 位值也是合法字段，但具体语义仍未完全确认

现阶段不能严谨断言这两个字段一定是速度、插值或偏移，只能确认它们属于明文格式的一部分，并且仅在前 16 个有效槽位内有意义。

## 4. 明文填充区的 8 字节块模式

如果把明文帧体按 8 字节分块，填充区会形成固定的三块循环：

1. `5555000000005555`
2. `0000000055550000`
3. `0000555500000000`

这组三块每帧重复 6 次，正好覆盖 144 字节 filler 区。

这件事对 `EYPT` 逆向很关键，因为 `EYPT` 文件也保留了对应的 8 字节块重复节奏。

## 5. EYPT 变体结构

仓库中只有 4 个文件使用 `EYPT`：

1. `1号前进.rob`
2. `2号后退.rob`
3. `3号单脚左转.rob`
4. `4号单脚右转.rob`

### 5.1 已证实的外层共性

`EYPT` 文件和明文文件共享同一个外层容器：

- 同样的 `ACT-40` 文件头
- 同样的帧数位置
- 同样的单帧长度 248 字节

### 5.2 已证实的帧内分段

`EYPT` 帧同样可以分成：

- 前 8 字节：变换后的帧头
- 后 240 字节：变换后的帧体

其中帧体又可以稳定分成两段：

- 前 96 字节：有效动作区
- 后 144 字节：固定尾部区

这个分段和明文文件的 `16 个有效槽位 + 24 个 filler 槽位` 完全对齐。

### 5.3 已证实的块级规律

把 `EYPT` 帧体按 8 字节分块后：

- 前 12 个块对应 96 字节活动区
- 后 18 个块对应 144 字节尾部区

后 18 个块在同一文件的所有帧中完全不变。

并且只会出现 3 种固定块值，各重复 6 次：

1. `0D0540292D558D23`
2. `5795378194AD14E6`
3. `38FFA6213AD74DCC`

这和明文 filler 区的三块循环一一对应，说明：
- `EYPT` 并没有改变逻辑内核
- 它只是把原本的 8 字节块做了私有变换

### 5.4 对 EYPT 的精确结论

在拿到上位机之后，现在已经可以把 `EYPT` 的保护层写成精确结论：

1. `EYPT` 是建立在同一个明文 `ACT-40` 内核之上的私有封装格式。
2. 上位机 `Bus Servo Control.exe` 的 `MainWindow` 类中直接存在：
   - `DecryptActionFile(byte[])`
   - `EncryptActionFile(byte[])`
   - `Decrypt(UInt16[])`
   - `Encrypt(UInt16[])`
   - `ENCRYPT_ARRAY : UInt32[]`
3. `ENCRYPT_ARRAY` 可以从程序集 `<PrivateImplementationDetails>` 中提取，当前样本中值为：
   - `0x00003D09`
   - `0x00000017`
   - `0x00001CCD`
   - `0x3B7B8488`
4. 8 字节块内部使用的是标准 `TEA`：
   - 分组长度：64 位
   - 密钥长度：128 位
   - 轮数：32
   - `delta = 0x9E3779B9`
5. `getU32Code(byte1, byte2, byte3, byte4)` 的拼装方式是小端：
   - `byte1 | (byte2 << 8) | (byte3 << 16) | (byte4 << 24)`
6. 每个 8 字节块独立处理，没有 `CBC`、`IV`、链式异或或额外混淆层。

### 5.5 EYPT 文件包装层的精确规则

`DecryptActionFile(byte[])` 和 `EncryptActionFile(byte[])` 的行为现在也已经明确：

1. 输入是完整 `.rob` 文件字节流。
2. 前 16 字节文件头不参与 `TEA` 计算。
3. 真正参与 `TEA` 的是 `data[16:]`，也就是完整帧区。
4. 帧区长度必须是 8 的倍数，否则上位机会报格式错误。
5. `DecryptActionFile(byte[])` 返回的是“不带 16 字节文件头的明文帧区”。
6. `EncryptActionFile(byte[])` 返回的是“不带 16 字节文件头的密文帧区”。

因此完整文件的重建规则是：

1. 明文文件 = 原文件头 + 解密后的帧区。
2. 把文件头 `8..11` 字节的 `EYPT` 标记清零，就得到普通明文 `ACT-40` 文件。
3. 重新加密时 = 明文文件头改回 `EYPT` + 加密后的帧区。

### 5.6 原始 UInt16[] 包装层的精确规则

`Encrypt(UInt16[])` 和 `Decrypt(UInt16[])` 不是直接对 `UInt16` 做 `TEA`，而是先做一层缓冲区抽取：

1. 先查找第一个值为 `0xFF00` 的 `UInt16`，它是终止哨兵。
2. 有效数据长度 = `哨兵索引 - 4`。
3. 前 4 个 `UInt16` 是控制前缀，不进入最终缓冲区。
4. 从第 5 个 `UInt16` 开始，只取每个元素的低 8 位，拷贝成字节流。
5. 字节流长度向上补齐到 8 的倍数，补零后再做 `TEA`。

也就是说，`UInt16[]` 版接口本质上只是“把带前缀和哨兵的 16 位数组，转换成字节缓冲区，再调用同一个 `TEA` 核心”。

### 5.7 EYPT 现状

现在已经解决：

1. `EYPT` 文件可以被稳定解密成明文 `ACT-40`。
2. `EYPT` 文件可以被稳定重新加密回原始密文。
3. `EncryptActionFile` / `DecryptActionFile` 的输入输出约定已经完全明确。
4. `Encrypt(UInt16[])` / `Decrypt(UInt16[])` 的缓冲区包装规则已经完全明确。
5. 保护层所用分组算法已经确认是标准 `TEA`，不是厂商自定义轮函数。

工程验证结果：

1. 仓库中的 4 个受保护官方动作，已经全部验证“独立解密 == 官方明文样本”。
2. 同样 4 个样本，已经全部验证“独立重加密 == 原始密文文件”。
3. `UInt16[]` 版 `Encrypt/Decrypt` 也已经与上位机方法逐字节对齐。

## 6. 当前已经确定的“内核”

如果只看逻辑内核，而不看 `EYPT` 的保护层，这套动作文件的核心结构已经很清楚：

1. 每个动作文件由多个固定长度帧组成。
2. 每帧都包含一个时长字段。
3. 每帧逻辑上只承载 16 个有效动作槽位。
4. 每个有效槽位由 3 个 16 位字段构成。
5. 后面 24 个槽位只是填充位。

也就是说，这个格式的真实动作内核不是“任意长度动作流”，而是：

- `时间 + 16 个槽位 * 3 个 16 位字段`

## 7. 当前仍未完全解决的点

下面这些点仍然属于“动作语义”问题，而不是“加密算法”问题：

1. 明文槽位中第 2、第 3 个 16 位字段的精确物理语义。
2. 明文帧头 8 字节里两个保留字段在控制层面的真实用途。

也就是说，`EYPT` 保护算法本身现在已经完全破解；剩下没完全定义的，只是明文动作字段的业务语义。

## 8. 工具支持

仓库里已经新增了三个脚本：

- `rob_reverse.py`
- `rob_decrypt_with_upper.ps1`
- `rob_crypto.py`

这些脚本目前可以做四件事：

1. 稳定解析明文 `.rob` 文件。
2. 输出 `EYPT` 文件的结构摘要和候选解码探针结果。
3. 通过上位机程序集直接把 `EYPT` 文件解密成明文 `.rob`。
4. 不依赖上位机，直接用独立 Python 实现对 `.rob` 做加解密。

用法示例：

- `./rob_decrypt_with_upper.ps1 -InputFile "动作\1号前进.rob"`

默认会生成：

- `动作\1号前进.plain.rob`

独立实现示例：

- `python rob_crypto.py decrypt-file "动作\1号前进.rob" "动作\1号前进.python.plain.rob"`
- `python rob_crypto.py encrypt-file "动作\1号前进.plain.rob" "动作\1号前进.python.rob"`

## 9. 最终结论

如果把“完整解析内核”理解为还原动作文件真正的数据骨架，那么这一步已经完成：

- 外层容器已确认
- 明文帧布局已确认
- 有效槽位数量已确认
- filler 结构已确认
- `EYPT` 与明文内核的对应关系已确认

如果把“完整解析”理解为“能拿到 `EYPT` 文件的真实明文动作数据”，这一步现在也已经完成。

如果把“完整解析”进一步提升为“把厂商私有变换写成独立的源码实现，而不再依赖上位机程序集”，这一步现在也已经完成。

当前可以明确写成最终结论：

1. `EYPT` 的真实保护算法是标准 `TEA-32`。
2. 保护对象是 16 字节文件头之后的完整帧区。
3. 文件头本身不加密，只通过 `8..11` 字节的 `EYPT` 标签区分明文和密文文件。
4. 仓库内已经有独立 Python 实现，并且与上位机和官方样本逐字节对齐。
