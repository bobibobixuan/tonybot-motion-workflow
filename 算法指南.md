# Tonybot 算法指南

## 1. 项目定位

这个项目是一个运行在 MicroPython 环境中的 Tonybot 控制程序。

程序的职责可以分成两层：

- 通信层：接收手机 APP 通过蓝牙发送的控制命令。
- 执行层：根据当前状态决定执行运动、动作组、避障或定距跟随。

动作本身不写在代码里，而是保存在 `动作` 目录下的 `.rob` 文件中。`main.py` 的作用是根据传感器和控制指令，决定在什么时机调用哪个动作组。

## 2. 整体结构

程序启动后，先完成硬件初始化：

- 创建机器人对象 `tonybot`
- 创建蓝牙对象 `ble`
- 创建 I2C 和超声波对象 `i2csonar`
- 设置默认灯光、头部角度和初始站立动作

之后并发启动两个主循环：

- `ble_receive()`：持续监听蓝牙命令，更新全局状态。
- `action_run()`：持续读取全局状态，执行对应动作和玩法。

可以把它理解成一个“输入线程 + 控制线程”的双循环模型。

## 3. 全局状态设计

程序大量依赖全局变量进行状态共享，核心变量如下：

- `status_motion`：当前手动运动指令。
- `status_action`：当前要执行的单次动作组编号。
- `status_func`：当前玩法模式。
- `_obs_step`：避障状态机当前步骤。
- `_walk_step`：定距跟随状态机当前步骤。
- `gDistance`、`gLDistance`、`gRDistance`：前方、左侧、右侧距离。
- `have_move`：记录当前是否已经进入运动过程。
- `lastActionIsGoBack`：记录上一次避障决策是否为后退。
- `l_stop`、`r_stop`：手动运动停止标志。
- `battery_volt`：缓存的电池电压。

这种设计的优点是简单直接，适合资源较紧张的嵌入式脚本；缺点是不同模块之间耦合较强，后续扩展玩法时需要小心状态冲突。

## 4. 动作映射

代码中先定义了一组常用动作编号：

- `actfirst = 18`：前进一步的过渡动作
- `actgo = 21`：循环前进
- `actback = 22`：循环后退
- `actleftskate = 11`：左侧滑
- `actrightskate = 12`：右侧滑
- `actturnleft = 23`：原地左转
- `actturnright = 24`：原地右转
- `actstandquickly = 19`：快速立正

然后通过 `action_list` 建立“控制命令 -> 动作编号”的索引关系：

- `1 -> 前进`
- `2 -> 后退`
- `3 -> 左侧滑`
- `4 -> 右侧滑`
- `5 -> 左转`
- `6 -> 右转`

这里的实现技巧是：`action_list[1]` 对应前进，`action_list[2]` 对应后退，因此列表的第 0 位只是为了让后面的索引和命令值对齐。

## 5. 蓝牙控制算法

`ble_receive()` 是通信入口，负责解析 APP 发来的 `CMD` 命令。

### 5.1 支持的命令类别

- 命令 `1`：手动运动控制
- 命令 `2`：运行指定动作组
- 命令 `3`：超声波相关功能
- 命令 `4`：玩法模式切换
- 命令 `5`：读取电池电量

### 5.2 手动运动控制

当命令类别为 `1` 时，程序会读取第二个参数作为运动命令：

- `1, 2, 3, 4`：前进、后退、左侧滑、右侧滑，同时清除 `l_stop`
- `5, 6`：左转、右转，同时清除 `r_stop`
- `8`：设置 `l_stop = True`，用于停止前后左右这组动作
- `9`：设置 `r_stop = True`，用于停止转向这组动作

最后统一写入 `status_motion`，供动作线程消费。

### 5.3 动作组控制

当命令类别为 `2` 时，直接把动作组编号写入 `status_action`，由调度线程执行一次。

### 5.4 超声波功能

当命令类别为 `3` 时：

- 参数为 `1`：读取超声波距离并通过蓝牙回传
- 参数为 `2`：设置超声波 RGB 灯颜色

这里距离值做了一个上限裁剪，最大回传 `500`，避免数值过大。

### 5.5 玩法模式切换

当命令类别为 `4` 时：

- `1`：进入避障模式
- `2`：进入定距跟随模式
- `0`：退出当前玩法模式

### 5.6 电量上报

当命令类别为 `5` 时，直接回传缓存中的 `battery_volt`。

## 6. 调度状态机

`action_run()` 是整个程序的控制核心。它通过局部变量 `step` 维护一个主状态机。

### 6.1 `step == 0`：空闲与分派

这是主入口状态，按优先级检查当前需求：

1. 如果有手动运动命令，转到 `step = 2`
2. 如果有动作组命令，转到 `step = 3`
3. 如果启用了避障玩法，转到 `step = 4`
4. 如果启用了定距跟随玩法，转到 `step = 5`
5. 如果都没有，则短暂休眠

这个优先级意味着：手动控制优先于自动玩法。

### 6.2 `step == 2`：执行手动运动

这里分成两类：

- `1, 2, 3, 4`：前后左右类动作，受 `l_stop` 控制
- `5, 6`：左右转类动作，受 `r_stop` 控制

算法特点：

- 每次并不是永久连续执行，而是执行一个动作周期后再回到循环检查停止条件。
- 前进和后退使用固定 `sleep` 时长控制节奏。
- 如果在前进或后退时收到停止指令，会补一个快速立正动作让姿态回正。

这种实现方式比较适合机器人动作组控制，因为动作文件通常是离散片段，不适合像电机那样持续给速度。

### 6.3 `step == 3`：执行单次动作组

逻辑最简单：

- 运行 `tmp_action`
- 等待执行结束
- 清空临时状态
- 回到 `step = 0`

### 6.4 `step == 4`：避障模式

进入避障模式时会先做一次初始化：

- 头部回到正中
- `_obs_step` 归零
- 执行 0 号基础动作

然后循环调用 `obstacleAvoidance()`。

如果玩法被关闭，则会：

- 停止当前动作组
- 执行站立动作
- 头部回正
- 返回主状态机空闲态

### 6.5 `step == 5`：定距跟随模式

进入定距跟随模式时会先做一次初始化：

- 设置蓝灯
- `_walk_step` 归零
- 执行 0 号基础动作

然后循环调用 `Distancewalking()`。

如果玩法关闭，则恢复默认灯效并回到空闲态。

### 6.6 周期性电量处理

`action_run()` 还做了两件后台任务：

- 每隔约 1 秒发送一次获取电量命令
- 每隔约 0.9 秒读取一次电量并缓存到 `battery_volt`

因此蓝牙读取电量时，APP 拿到的不是实时硬件直读值，而是最近一次缓存结果。

## 7. 避障算法详解

避障逻辑由 `obstacleAvoidance()` 完成，是一个四步状态机。

### 7.1 参数含义

- `MIN_DISTANCE_TURN = 200`：小于这个值视为前方空间不足，需要考虑转向或后退
- `BIAS = 0`：头部舵机角度偏移，用于校准

距离单位在代码里统一乘以 10，便于处理整数值。

### 7.2 `_obs_step == 0`：前方探测

流程如下：

1. 读取正前方距离 `gDistance`
2. 如果前方距离足够大，或者传感器返回 0，则认为可以前进
3. 若机器人当前没有在运行，就执行前进一步过渡动作，再切到循环前进动作，并进入 `_obs_step = 1`
4. 如果前方距离过近，则直接进入 `_obs_step = 2`，准备扫描周围环境

这里把距离为 0 视作“可走”，实际含义通常是超声波没有测到有效障碍物。

### 7.3 `_obs_step == 1`：监控前进过程

机器人已经在前进时，持续检测前方距离：

- 如果前方仍然安全，就保持当前过程
- 一旦发现前方距离小于阈值，就执行一次停止收尾动作，然后进入 `_obs_step = 2`

这一步的作用是避免机器人在持续前进时撞上突然靠近的障碍物。

### 7.4 `_obs_step == 2`：环境扫描

当机器人停止后，如果当前没有动作在运行，就调用 `getAllDistance()` 采集多方向距离：

- 正前方：`90 + BIAS`
- 左前方过渡角：`145 + BIAS`
- 左侧：`180 + BIAS`
- 右前方过渡角：`45 + BIAS`
- 右侧：`0 + BIAS`

算法不是简单记录左右两个单点，而是把过渡角和侧向角做一次最小值比较，从而让左右距离更保守：

- 左侧距离 = `min(左前方过渡角, 左侧)`
- 右侧距离 = `min(右前方过渡角, 右侧)`

这样做可以减少“侧边看起来很空，但斜前方其实有障碍”的误判。

### 7.5 `_obs_step == 3`：决策转向或后退

这是避障算法的决策核心：

1. 如果前方已经重新变得安全，而且上一次不是后退导致的局部恢复，则直接回到 `_obs_step = 0`
2. 如果左侧比右侧更宽敞，且左侧距离足够，就优先左转
3. 如果右侧比左侧更宽敞，且右侧距离足够，就优先右转
4. 如果左右都不合适，则执行后退策略

具体动作策略如下：

- 已经处于前进过程时，先执行左转或右转过渡动作 `36` 或 `37`
- 然后执行原地转向动作 4 次
- 如果两侧都不满足条件，则执行前进一步过渡、后退两步、再恢复站立

`lastActionIsGoBack` 的作用是防止机器人在刚后退完时，因为前方暂时变远就立刻又进入前进，导致来回抖动。

### 7.6 避障算法特点总结

这个避障算法不是地图式规划，而是基于规则的局部反应式控制，特点是：

- 简单，适合资源有限的单片机环境
- 响应快，不依赖复杂建图
- 可解释性强，便于调阈值
- 容易在狭窄环境中出现局部反复试探

## 8. 定距跟随算法详解

`Distancewalking()` 也是一个状态机，用于根据人与机器人的距离决定前进、后退或停止。

### 8.1 距离区间设定

- `30 < Distance < 180`：目标太近，机器人后退
- `300 < Distance < 400`：目标太远，机器人前进
- 其他区间：机器人停止并保持待机

这意味着中间的舒适距离带比较宽，机器人不会频繁抖动。

### 8.2 `_walk_step == 0`：判定要不要动作

这是待机态：

- 如果目标太近，亮红灯，执行前进一步过渡动作，然后进入 `_walk_step = 1`
- 如果目标太远，亮绿灯，执行前进一步过渡动作，然后进入 `_walk_step = 2`
- 如果之前已经移动过但现在不该继续移动，则进入 `_walk_step = 3`
- 如果没有动作需求，则亮蓝灯等待

### 8.3 `_walk_step == 1`：执行后退

如果目标仍然太近，或者标志位允许继续动作，则：

- 执行一次后退动作
- 等待动作结束

如果距离已经恢复到正常范围，则进入 `_walk_step = 3` 做收尾。

### 8.4 `_walk_step == 2`：执行前进

如果目标仍然太远，或者标志位允许继续动作，则：

- 执行一次前进动作
- 等待动作结束

如果距离恢复正常，则进入 `_walk_step = 3`。

### 8.5 `_walk_step == 3`：收尾复位

收尾阶段会：

- 执行过渡动作
- 执行快速立正
- 恢复蓝灯
- 清除 `have_move`
- 状态回到 `_walk_step = 0`

这一步的作用是保证机器人从移动姿态平滑恢复到标准站立姿态。

## 9. 灯光在算法中的作用

超声波模块 RGB 灯不仅是装饰，也被用作状态反馈：

- 绿色：前方可行或需要前进
- 红色：目标过近，需要后退
- 蓝色：空闲或等待
- 青色：正在采集多方向距离

这种设计对调试很有帮助，因为即使不看串口日志，也能从灯光快速判断当前状态机处于哪个阶段。

## 10. 这个算法适合什么场景

适合：

- 教学演示
- APP 遥控机器人
- 简单室内避障
- 面向动作组的行为编排

不适合：

- 高速运动
- 复杂动态障碍环境
- 需要路径规划和地图记忆的场景
- 需要多传感器融合的稳定跟随

## 11. 调参建议

如果你要继续调这个项目，优先关注这几个参数：

- `MIN_DISTANCE_TURN`：影响避障触发灵敏度
- `BIAS`：影响测距方向是否居中
- `Distancewalking()` 中的距离区间：影响跟随距离感受
- 各个 `sleep()` 和 `waitForStop()` 时长：影响动作节奏和稳定性

推荐的调参顺序：

1. 先校准头部和超声波方向
2. 再确定避障阈值
3. 最后微调前进、后退、转向的动作时长

## 12. 后续优化方向

如果后面要继续维护这个项目，可以优先做下面几件事：

1. 把全局变量封装成一个状态对象，降低耦合。
2. 把蓝牙协议解析、动作调度、玩法算法拆成独立模块。
3. 给动作命令定义常量或枚举，避免代码里直接写数字。
4. 把避障和跟随阈值抽成配置项，便于调试。
5. 给不同状态机补充日志输出，方便定位问题。

## 13. 一句话理解这个程序

这个程序的核心思想是：

“蓝牙负责改状态，主循环负责读状态，动作组负责执行动作，超声波负责提供局部环境反馈。”

只要理解了这四层关系，就能很快读懂整个项目。
